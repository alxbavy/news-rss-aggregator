# News RSS Aggregator

Приложение опрашивает RSS-ленты новостных сайтов, сохраняет новые записи в базу данных и отправляет уведомления в Telegram.  
Проект реализован как небольшой backend-сервис на FastAPI с отдельным scheduler-процессом.

## Что делает приложение

- хранит список RSS-источников в базе данных;
- хранит список Telegram-чатов для рассылки;
- опрашивает активные RSS-ленты;
- извлекает из ленты заголовок, ссылку, дату публикации и вычисляет хэш контента;
- определяет, какие посты уже были обработаны ранее;
- сохраняет только новые посты;
- отправляет уведомления в Telegram по новым постам.

## Архитектура проекта

Проект разделён на несколько слоёв:

- `app/api` - HTTP API
- `app/admin` - HTML-админка
- `app/services` - use-case логика
- `app/repositories` - работа с БД
- `app/integrations` - внешние интеграции, RSS parser и telegram notifier
- `app/models` - SQLAlchemy-модели
- `app/schemas` - Pydantic-схемы
- `app/scheduler` - периодический запуск polling-процесса

### Схема взаимодействия компонентов

```text
[Scheduler]
    |
    v
[FeedPollingService]
    |-----------------------> [RSSParser] ---> получает записи с новостных сайтов
    |
    |-----------------------> [Storage] ---> управление БД
    |
    |-----------------------> [TelegramNotifier] ---> отправляет сообщения в Telegram
    |
    v
[PostgreSQL]
```

## Логика дедупликации

В качестве основы для дедупликации выбран персистентный кеш в PostgreSQL.
Хранение в памяти процесса не подошло бы, потому что после перезапуска приложения эта информация пропадёт.  

Внутри парсинга одной ленты дополнительно используется временное множество `seen_hashes`.  
Оно нужно, чтобы отсеять дубликаты, которые иногда встречаются внутри одной RSS-ленты за один запрос.  

Таким образом, в проекте используется двухуровневый подход, временный in-memory фильтр - против дублей внутри текущего цикла, постоянное хранение в БД - против дублей между запусками и перезапусками.
